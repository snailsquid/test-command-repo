## ADDED Requirements

### Requirement: Developer registration via WhatsApp
The system SHALL allow developers to register by sending `.register <token>` to any Akka contact. The token is generated by the web portal.

#### Scenario: Generate registration token
- **WHEN** a developer visits the web portal and clicks "Register"
- **THEN** the system SHALL generate a unique random token and display it

#### Scenario: Register via WhatsApp
- **WHEN** a developer sends `.register abc123` to any contact
- **THEN** the system SHALL verify the token and link the developer's WhatsApp JID to their account
- **THEN** the system SHALL confirm registration via WhatsApp message

#### Scenario: Invalid token
- **WHEN** a developer sends `.register invalidtoken`
- **THEN** the system SHALL respond that the token is invalid or expired

### Requirement: Developer authentication
The system SHALL authenticate developers before granting access to the portal.

#### Scenario: Developer login
- **WHEN** a developer provides valid credentials
- **THEN** the system SHALL grant access to the developer dashboard

#### Scenario: Invalid credentials
- **WHEN** a developer provides invalid credentials
- **THEN** the system SHALL deny access and show an error

### Requirement: Command registration
The system SHALL allow developers to register new commands by providing a GitHub repository URL and metadata.

#### Scenario: Register command
- **WHEN** a developer fills in the command registration form (slug, name, description, usage, GitHub URL)
- **THEN** the system SHALL fetch and validate the command code
- **THEN** the system SHALL add the command to the marketplace under the developer's ID

### Requirement: Developer group management
The system SHALL allow developers to create and manage developer groups.

#### Scenario: Create developer group
- **WHEN** a developer creates a new group with a name
- **THEN** the system SHALL create the group and add the creator as a member

#### Scenario: Add member to group
- **WHEN** a developer adds another developer to their group
- **THEN** the added developer SHALL have equal access to all group commands

### Requirement: Command analytics
The system SHALL display usage analytics for commands owned by the developer or their group.

#### Scenario: View analytics
- **WHEN** a developer views a command's analytics
- **THEN** the system SHALL show: total installs, installs by contact, usage count, error count

### Requirement: Command management
The system SHALL allow developers to update or disable commands they own.

#### Scenario: Disable command
- **WHEN** a developer disables a command
- **THEN** the command SHALL be removed from the marketplace
- **THEN** existing installations SHALL stop working

#### Scenario: Update command
- **WHEN** a developer updates the GitHub URL or metadata
- **THEN** the system SHALL re-fetch and validate the command code

### Requirement: Group-scoped command visibility
The system SHALL only show commands to developers that belong to their group or that they own individually.

#### Scenario: Developer views commands
- **WHEN** a developer views the command list
- **THEN** they SHALL only see commands owned by them or by groups they belong to
